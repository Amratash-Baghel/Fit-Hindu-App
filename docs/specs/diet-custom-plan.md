# Spec — Diet section + AI custom plan

> Status: owner-directed 2026-07-16. **This feature is an explicit owner
> override** of the standing "no AI plan generation" v1 rule (CLAUDE.md;
> decisions 2026-07-10; onboarding spec). The override is scoped to the diet
> custom-plan feature only — the onboarding plan engine stays rule-based.

## Purpose

Give users a personalised diet plan generated by AI from a short questionnaire,
delivered into the app. Adds a Diet tab that also surfaces the admin's
rule-based diet templates.

## Two questionnaires (do not confuse)

- **Onboarding questionnaire** (`docs/specs/onboarding-questionnaire.md`) —
  rule-based, no AI, no free-text. Unchanged.
- **Diet custom-plan questionnaire** (this spec) — collects ht/wt/region +
  goal/diet/activity and feeds the AI pipeline. Separate flow, separate intent.

## User flow

1. **Diet tab** (`app/(tabs)/diet.tsx`): a "Generate your custom diet plan" hero
   CTA + the published rule-based diet templates + (if present) a "Your custom
   plan" card linking to the latest ready plan.
2. **Questionnaire** (`app/diet/questionnaire.tsx`): height (cm), weight (kg),
   region (North/South/East/West/Central/North-East Indian), goal, diet type,
   activity level. Kept deliberately short.
3. On submit (`src/lib/diet.ts::submitDietRequest`):
   - insert a `diet_plan_requests` row (`status='pending'`, answers jsonb);
   - mirror height/weight/region onto `profiles`;
   - POST `{request_id, answers}` to the n8n webhook (best-effort).
4. **Plan screen** (`app/diet/plan.tsx`): polls the row every 4s; renders
   `generating` (preparing state) → `ready` (structured plan) → `failed`
   (retry). Carries the AI wellness disclaimer.

## Backend

- Migration `0010`: `diet_plan_requests` (own-row RLS: read+insert only — the
  plan is written back by n8n via the service-role key, which bypasses RLS, so
  clients can't forge a plan), new enum `diet_request_status`, and profile
  columns `height_cm/weight_kg/region/body_focus/days_per_week`.
- n8n workflow `n8n/diet-plan-workflow.json` (Anthropic Claude) + setup guide
  `docs/n8n-diet-setup.md`. The app holds only the webhook URL + a shared
  secret (`EXPO_PUBLIC_N8N_DIET_WEBHOOK_URL` / `EXPO_PUBLIC_N8N_DIET_SECRET`);
  the Supabase service-role key and the Anthropic key live only in n8n.
- Plan shape: typed `DietPlan` in `src/types/db.ts`; the AI is instructed to
  return exactly that JSON.

## Rules / guardrails

- AI output is **general wellness only** — never medical advice or cure claims.
  The system prompt enforces this and the app shows a disclaimer on both the
  questionnaire and the plan.
- Diet type (veg/sattvic/egg/nonveg) must be respected strictly by the AI.
- User-owned data → requires auth (RLS). Until app auth ships, submit returns
  `{ok:false, reason:'auth'}` and the UI shows the "activates with sign-in"
  note — same staged state as the rest of the app.

## Compliance follow-ups (required before launch)

New AI processor (Anthropic) + new health data (ht/wt/region) must be added to
`legal/THIRD_PARTY_PROCESSORS.md`, `legal/DATA_INVENTORY.md`, the privacy
policy, and the Play Data Safety form. Tracked in `docs/decisions.md`
(2026-07-16).

## Not doing (this iteration)

- No admin review/approval queue (owner chose AI-direct-to-user).
- No saving the generated plan into the rule-based `user_plans` engine — the AI
  plan lives in `diet_plan_requests.plan` and is shown directly.
- No editing the AI plan in-app.
